# EXMC（Exchange Monte Carlo）実装方針

`exchange_mc_mpi_tips.md` に従い、`classical-mc-simple` に Exchange MC（並列テンパリング）と MPI を段階的に追加する実装方針をまとめる。

---

## 1. 現状の整理

- **データ構造**: 既に `int_T` ごとの配列がある
  - `Def.sx[int_T][all_i]`, `Def.sy`, `Def.sz`
  - `Def.env_sx[int_T][all_i]`, `Def.env_sy`, `Def.env_sz`
  - `Phys.Energy[int_T]`, `Phys.ratio_1[int_T]`
- **メインループ**: 各ステップで「全 `int_T` に対して 1 sweep の MC」→ 観測
- **温度**: `T = Ini_T + Delta_T * int_T`（1 プロセス時）

この「温度インデックス × 状態」の構造をそのまま活かして EXMC を載せる。

---

## 2. EXMC の受諾確率（そのまま利用）

隣接する温度インデックス `i`, `j` のレプリカを入れ替える試行の受諾確率:

```text
P_accept = min(1, exp((1/T_i - 1/T_j) * (E_i - E_j)))
```

詳細釣り合いを満たす標準形。実装時は `T_i, T_j` と `Energy[i], Energy[j]` を使う。

---

## 3. 実装ステージ（tips の 4 に沿った段階的計画）

### Stage A: 単一プロセス内 EXMC（MPI なし）

**目的**: デバッグを簡単にし、エネルギー・観測量の温度連続性を確認する。

1. **交換試行の追加**
   - 各 MC ステップで「全 `int_T` に対して 1 sweep」の**後**に、隣接温度ペア `(int_T, int_T+1)` の交換を試行する。
   - 受諾時: レプリカ `int_T` と `int_T+1` の以下を**丸ごとスワップ**する。
     - `sx[int_T][]` ↔ `sx[int_T+1][]`（sy, sz も同様）
     - `env_sx[int_T][]` ↔ `env_sx[int_T+1][]`（env_sy, env_sz も同様）
     - `Phys.Energy[int_T]` ↔ `Phys.Energy[int_T+1]`
   - `ratio_1` は「その温度スロットで Metropolis が何回受諾されたか」のカウンタなので、**交換では変更しない**（温度と状態が入れ替わるだけ）。

2. **交換のループ順（デッドロック回避の準備）**
   - 単一プロセスでは 1 本のループでよい。隣接ペアを例えば `(0,1), (2,3), (4,5), ...` と組ませ、次に `(1,2), (3,4), ...` と組ませる「odd-even」で試行すると、のちの MPI 版と揃えやすい。

3. **観測**
   - 交換の有無にかかわらず、「現在その温度スロットにいる状態」で E, M^2 等を計測する（既存の `Energy[int_T]` と `magnetization_sq(..., int_T)` をそのまま利用）。

4. **確認すべきこと**
   - 温度 T に対する E/N, C, M^2 の曲線が滑らかにつながる。
   - 交換受諾率が極端に 0 や 1 に偏っていない（目安: 0.1〜0.5 程度が無難）。

---

### Stage B: 温度インデックスのグローバル化（MPI 準備）

**目的**: 複数 rank を扱うときの温度の一意な割り当てを決める。

1. **グローバル温度インデックス**
   - `gT = rank * num_temp + int_T`
   - 温度は `T(rank, int_T) = Ini_T + Delta_T * gT`
   - 全 rank で `num_temp` を「1 rank あたりの温度数」とし、総温度数 = `total_proc * num_temp` としてもよいし、最初は「1 rank が全温度を持つ」形で Stage A と等価にしてもよい。

2. **実装上の方針**
   - `Phys.T` は「現在の `int_T` に対応する温度」なので、単一 rank のときは従来通り `Ini_T + Delta_T * int_T`。
   - MPI 版では、その rank が担当する `int_T` に対応する `gT` から `T = Ini_T + Delta_T * gT` を計算するようにする。
   - パラメータ設計: 総温度数や `Ini_T`, `Delta_T` を「全レプリカ」用に 1 通りで定義し、各 rank は自分の `(rank, int_T)` に対応する `gT` だけ使う。

---

### Stage C: MPI プロセス topology と境界交換

**目的**: 複数 rank でレプリカを分割し、rank 間で隣接温度の交換を行う。

1. **担当温度の分割**
   - 各 rank が `num_temp` 個の温度スロットを担当。
   - rank `r` が持つ温度（グローバル）は `gT = r * num_temp + int_T`（int_T = 0 .. num_temp-1）。

2. **交換の種類**
   - **rank 内**: 隣接する `int_T` と `int_T+1` の交換（Stage A と同じ）。
   - **rank 間**: 隣接する `gT` が別 rank にある場合、その rank と「境界」の状態・エネルギーを交換する。
     - 例: rank 0 の最大 `gT` と rank 1 の最小 `gT` のペアで交換試行。

3. **デッドロック回避（odd-even pairing）**
   - 全ペアを「偶数 gT と奇数 gT」のように 2 グループに分ける。
   - 片方のグループだけを一斉に交換試行するとき、送信/受信の順序を「偶数 rank が先に send、奇数 rank が先に recv」などと揃える（またはその逆）。これで全 rank が同じ順序で MPI_Send/Recv を呼べばデッドロックを防げる。
   - 実装では「境界交換するペア」を列挙し、そのペア間で (E_i, E_j) と状態（sx,sy,sz のバッファ）を交換し、受諾判定後に必要なら状態を上書きする。

4. **バッファ**
   - `Def.send_buffer`, `Def.recv_buffer` は既に `mc_def.h` に存在。交換するスピン（と必要なら env）をパックして送り、受諾時は受け取ったデータで上書きする。
   - 非受諾の場合は「送った側」は自分の状態をそのまま、「受け取った側」も自分の状態をそのまま維持。

5. **ローカル有効場 `env_*` の扱い**
   - 交換で「状態」を入れ替えた場合、その温度スロットの `env_*` は**その状態から再計算**する必要がある（他 rank から来た状態に対して、現在の格子の Nr, J で env を一から計算する）。さもないと `env_*` が古い状態のままになり、次の Metropolis が破綻する。
   - あるいは、交換時に `env_*` も一緒に送受信する（データ量は増えるが、再計算コストを避けられる）。格子は全 rank で同じなので、env をそのまま使ってよい。

---

### Stage D: テストとチューニング

1. **再現性**
   - 各 rank で異なる RNG シード（例: `base_seed + 8945UL * int_samp + 12345UL * rank`）を使う。同じシードにすると bias や相関の原因になる。

2. **温度間隔**
   - `Delta_T` が大きすぎると交換受諾率がほぼ 0 になる。目安として受諾率が 0.1〜0.5 になるよう、温度点数や間隔を調整する。

3. **観測**
   - 各温度（gT）ごとに、そのスロットに「今いる」状態の E, M^2 等を記録。複数サンプル・複数ステップで平均し、熱力学的な曲線が滑らかか確認する。

---

## 4. 実装時の注意（tips の 5 をコードに落とす）

| 項目 | 対策 |
|------|------|
| 温度インデックスのずれ | `gT = rank * num_temp + int_T` を一箇所で計算する関数/マクロにし、T の計算と交換ペアの決定の両方でそれを使う。 |
| デッドロック | 境界交換は odd-even などで送受信順を全 rank で一致させる。 |
| 交換後の env 不整合 | 交換受諾時に env を再計算するか、env ごと送受信して上書きする。 |
| RNG シード | 各 rank で異なるシード（rank を加える）。 |
| 受諾率が 0 に近い | 温度点数を増やす、または Delta_T を小さくする。 |

---

## 5. ファイル・関数の役割（案）

- **mc_def.h**
  - 既存の `send_buffer`, `recv_buffer` を EXMC の状態パック用に使用。必要ならサイズや「パック/アンパック」の仕様を明文化。
- **mc_update.c**
  - `MC()` はそのまま（1 温度スロット 1 sweep）。
  - 新規: `exchange_try_pair(X, i, j)` … 温度スロット i と j の交換を 1 回試行（受諾確率の計算、乱数、受諾時は sx/sy/sz/env/Energy のスワップ、および必要なら env 再計算または env スワップ）。
  - 新規: `exchange_sweep_local(X)` … rank 内の隣接ペアに対して `exchange_try_pair` を odd-even で呼ぶ。
- **main.c（または exmc_mpi.c のような別ファイル）**
  - Burn-in / Measurement ループで「各 int_T で MC() → exchange_sweep_local() → （MPI 時）exchange_sweep_boundary() → 観測」の順に実行。
  - MPI 初期化、rank/total_proc の設定、グローバル温度 `gT` と T の対応。
- **lattice.c**
  - 既存の `initial()` は全 `int_T` を初期化するので、EXMC 単体ではそのまま利用可能。MPI 時は「この rank が担当する int_T だけ初期化」に変更するか、あるいは全 rank が同じ格子・同じ num_temp で自分のスロットだけ初期化する形にする。
- **memory.c**
  - 1 rank あたり `num_temp` スロットを維持する現状の割り当てのまま。MPI で「総温度数 = total_proc * num_temp」にする場合は、各 rank の `num_temp` をどう解釈するか（「自分のスロット数」）を統一する。

---

## 6. まとめ

1. **まず Stage A**: 単一プロセスで隣接温度の交換だけ追加し、データフローと観測量を確認する。
2. **Stage B**: 温度を `gT` で表すようにし、のちに rank ごとの担当範囲と対応させる。
3. **Stage C**: MPI の rank 間で境界の交換を odd-even で実装し、交換後の `env_*` を必ず整合させる。
4. **Stage D**: シード・温度間隔・受諾率を調整し、熱力学的な曲線と再現性を確認する。

この順で進めれば、`exchange_mc_mpi_tips.md` の「既存の mc_simple のデータフローを活かす」「まず intra-process EXMC」「デッドロック・env の不整合・シード・温度間隔に注意」という方針に沿った実装になる。
